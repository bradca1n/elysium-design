# Copper Custodian Integration Design

> Detailed design for integrating Copper.co crypto custody into the Elysium fund administration platform. Covers the `@elysium/copper` SDK package, the custody-ops service, and all operational workflows.

**Goal:** Enable fully automated custody operations — deposit tracking, internal settlement transfers, and external withdrawals — through Copper's API, with zero manual intervention for standard operations.

**Depends on:** [Elysium Platform Architecture](./2026-03-05-elysium-platform-architecture.md) for EventBridge, Step Functions, observability, and resilience patterns.

---

## 1. Copper API Summary

### 1.1 Authentication

Copper uses HMAC-SHA256 request signing (no Bearer tokens):

```
ApiKey: <api-key>
Signature: HMAC-SHA256(secret, timestamp + method + path + body)
Timestamp: <unix-epoch-ms>
```

Every request requires all three headers. The signature covers the full request body, preventing tampering.

### 1.2 Core API Endpoints

| Operation | Method | Path | Purpose |
|-----------|--------|------|---------|
| List portfolios | GET | `/platform/portfolios` | List all portfolios (= fund accounts) |
| Create portfolio | POST | `/platform/portfolios` | Create new fund account |
| Get balances | GET | `/platform/portfolios/{id}/balances` | Current balances per portfolio |
| Create order | POST | `/platform/orders` | Unified transfer/withdrawal API |
| Get order | GET | `/platform/orders/{id}` | Check order status |
| List orders | GET | `/platform/orders` | Query orders with filters |
| Create deposit target | POST | `/platform/deposit-targets` | Generate deposit addresses |
| Get deposit target | GET | `/platform/deposit-targets/{id}` | Get address for deposits |
| List transactions | GET | `/platform/transactions` | Historical transaction log |
| Address book | POST | `/platform/addresses` | Register external withdrawal addresses |
| Get address | GET | `/platform/addresses/{id}` | Look up registered address |
| Webhooks | POST | `/platform/webhooks` | Register webhook endpoints |

### 1.3 Key Concepts

**Portfolios** — Copper's equivalent of accounts/sub-accounts. Each Elysium fund maps to one Copper portfolio. A "Cash" portfolio holds unallocated investor deposits.

**Orders** — The unified API for all transfers. Order types:
- `withdraw` — send crypto from portfolio to external address
- `transfer-out` / `transfer-in` — move between portfolios within the same organization
- `convert` — currency conversion (future use)

**Deposit Targets** — Blockchain addresses generated by Copper for receiving deposits. Each target is linked to a portfolio and a specific currency/network.

**Address Book** — Pre-registered external addresses for withdrawals. Copper can enforce that withdrawals only go to addresses in the Address Book (compliance).

**Webhooks** — Copper sends POST requests for order status changes, deposit confirmations, and settlement completions.

### 1.4 Environments

| Environment | Base URL | Purpose |
|-------------|----------|---------|
| Staging | `https://api.staging.copper.co/platform` | Integration testing |
| Production | `https://api.copper.co/platform` | Live operations |

### 1.5 Rate Limits

- 10 requests/second per API key (standard tier)
- Bulk operations should use pagination and backoff
- Rate limit headers: `X-RateLimit-Remaining`, `X-RateLimit-Reset`

### 2.2 CopperClient

```typescript
interface CopperClientConfig {
  baseUrl: string          // https://api.copper.co/platform or staging
  apiKey: string           // Copper API key
  apiSecret: string        // HMAC signing secret
  timeoutMs?: number       // Default: 30_000
  maxRetries?: number      // Default: 3
  logger?: Logger          // @elysium/logger instance
}

class CopperClient {
  constructor(config: CopperClientConfig)

  // Signed HTTP methods — all requests auto-sign with HMAC-SHA256
  get<T>(path: string, params?: Record<string, string>): Promise<T>
  post<T>(path: string, body: unknown): Promise<T>
  put<T>(path: string, body: unknown): Promise<T>
  delete<T>(path: string): Promise<T>

  // Rate limit tracking
  getRateLimitRemaining(): number
}
```

**HMAC Signing Implementation:**

```typescript
function signRequest(
  secret: string,
  timestamp: string,
  method: string,
  path: string,
  body: string
): string {
  const message = timestamp + method.toUpperCase() + path + body
  return crypto.createHmac('sha256', secret).update(message).digest('hex')
}
```

### 2.7 Webhook Verification

```typescript
// Verify webhook signature from Copper
function verifyWebhookSignature(
  payload: string,
  signature: string,
  secret: string
): boolean {
  const expected = crypto.createHmac('sha256', secret).update(payload).digest('hex')
  return crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(expected))
}
```

## 3. Service: Custody Operations

The custody-ops service is NOT a standalone microservice — it's a set of Lambda functions, Step Functions workflows, and EventBridge rules deployed via Terraform alongside the existing API infrastructure.

### 3.1 Component Map

```
EXTERNAL                    AWS                               INTERNAL
─────────                   ───                               ────────
Copper Webhook ──→ API Gateway ──→ Lambda: webhook-handler
                                       │
                                       ▼
                                   EventBridge (elysium-events)
                                       │
                                       ├──→ Step Functions: deposit-processing
                                       ├──→ Step Functions: settlement-processing
                                       ├──→ Step Functions: offramp-processing
                                       └──→ Step Functions: fund-provisioning
                                                    │
                                                    ▼
                                               Lambda: step handlers
                                                    │
                                   ┌────────────────┼─────────────────┐
                                   ▼                ▼                 ▼
                              Copper API       RDS (Postgres)    EventBridge
                              (@elysium/       (state, audit)    (emit events)
                               copper)
```

### 3.2 Database Schema (Additions to existing RDS)


## 4. Workflows

### 4.1 Deposit Processing

**Trigger:** Copper webhook fires `deposit.confirmed` event.

```
┌─────────────────────────────────────────────────────────┐
│                DEPOSIT PROCESSING WORKFLOW                │
├──────────┬──────────────────────────────────────────────┤
│  Step 1  │  WEBHOOK RECEIVED                             │
│          │  Copper sends deposit.confirmed webhook        │
│          │  → Lambda verifies signature                   │
│          │  → Emits DEPOSIT_DETECTED event                │
│          │  → EventBridge rule starts this Step Function  │
├──────────┼──────────────────────────────────────────────┤
│  Step 2  │  IDENTIFY SENDER                              │
│          │  → Query investor_wallets by sender address    │
│          │  → If found: attach investorId                 │
│          │  → If NOT found: flag for manual review,       │
│          │    emit UNIDENTIFIED_DEPOSIT alert              │
├──────────┼──────────────────────────────────────────────┤
│  Step 3  │  VALIDATE DEPOSIT                             │
│          │  → Check currency is supported (USDC/USDT)    │
│          │  → Check amount > minimum threshold            │
│          │  → Check investor is active and KYC'd          │
├──────────┼──────────────────────────────────────────────┤
│  Step 4  │  RECORD DEPOSIT                               │
│          │  → Write custody_operation record              │
│          │  → Emit DEPOSIT_CONFIRMED event                │
│          │  → Notify investor (via lifecycle event)       │
├──────────┼──────────────────────────────────────────────┤
│  Step 5  │  COMPLETE                                     │
│          │  → Cash now available in Cash portfolio        │
│          │  → Investor can submit subscribe order         │
└──────────┴──────────────────────────────────────────────┘
```

**Deposit arrives at the fund's shared Copper deposit address.** Sender identification works by matching the blockchain `from` address against the `investor_wallets` table. Investors must register their wallet before depositing.

### 4.2 Settlement Processing (Subscribe/Redeem)

**Trigger:** EventBridge event `ORDER_BATCH_PROCESSED` from order-processor.

When the on-chain order book processes subscribe/redeem orders, the custody side must move assets between portfolios:
- **Subscribe:** Cash portfolio → Fund portfolio (investor buying fund shares)
- **Redeem:** Fund portfolio → Cash portfolio (investor selling fund shares)

```
┌──────────────────────────────────────────────────────────┐
│              SETTLEMENT PROCESSING WORKFLOW                │
├──────────┬───────────────────────────────────────────────┤
│  Step 1  │  PARSE ORDER BATCH                             │
│          │  → Extract individual subscribe/redeem orders   │
│          │  → Map fund_id → Copper portfolio_id            │
│          │  → Calculate net amounts per fund               │
├──────────┼───────────────────────────────────────────────┤
│  Step 2  │  SUBMIT TRANSFERS (per order)                  │
│          │  → POST /platform/orders (transfer)             │
│          │  → externalOrderId = {fundId}-{dealingId}-{orderId}│
│          │  → Store taskToken keyed by copperOrderId       │
│          │  → WAIT FOR CALLBACK                            │
├──────────┼───────────────────────────────────────────────┤
│  Step 3  │  CALLBACK RECEIVED                             │
│          │  → Copper webhook fires order.settled           │
│          │  → Webhook Lambda calls sfn.sendTaskSuccess()   │
│          │  → Workflow resumes                             │
├──────────┼───────────────────────────────────────────────┤
│  Step 4  │  CONFIRM SETTLEMENT                            │
│          │  → Verify Copper order status = 'settled'       │
│          │  → Update custody_operation status              │
│          │  → Emit SETTLEMENT_COMPLETED event              │
├──────────┼───────────────────────────────────────────────┤
│  Step 5  │  ALL ORDERS SETTLED                            │
│          │  → Emit DEALING_SETTLED event                   │
│          │  → Triggers any downstream processing           │
└──────────┴───────────────────────────────────────────────┘
```

**Handling multiple orders:** Step Functions Map state iterates over the order array. Each order is an independent transfer submitted in parallel (Copper processes them independently). The Map state waits for all to complete.

### 4.3 Offramp Processing (Withdrawal)

**Trigger:** API request from admin/manager, or automated after redeem settlement.

When an investor redeems and wants to withdraw to their external wallet:

```
┌──────────────────────────────────────────────────────────┐
│               OFFRAMP PROCESSING WORKFLOW                  │
├──────────┬───────────────────────────────────────────────┤
│  Step 1  │  VALIDATE WITHDRAWAL                           │
│          │  → Check investor has sufficient cash balance   │
│          │  → Check destination address is in Address Book │
│          │  → Check amount > minimum, < daily limit        │
│          │  → Check investor is not frozen/suspended       │
├──────────┼───────────────────────────────────────────────┤
│  Step 2  │  SUBMIT WITHDRAWAL                             │
│          │  → POST /platform/orders (withdraw)             │
│          │  → externalOrderId = {fundId}-offramp-{investorId}-{timestamp}│
│          │  → Copper-native approval policies apply        │
│          │  → Store taskToken keyed by copperOrderId       │
│          │  → WAIT FOR CALLBACK                            │
├──────────┼───────────────────────────────────────────────┤
│  Step 3  │  APPROVAL (Copper-native)                      │
│          │  → Copper's policy engine evaluates             │
│          │  → May require manual approval for large amounts│
│          │  → Webhook fires when approved or rejected      │
├──────────┼───────────────────────────────────────────────┤
│  Step 4  │  EXECUTION                                     │
│          │  → Copper broadcasts blockchain transaction     │
│          │  → Webhook fires on-chain confirmation          │
│          │  → sfn.sendTaskSuccess() resumes workflow       │
├──────────┼───────────────────────────────────────────────┤
│  Step 5  │  CONFIRM OFFRAMP                               │
│          │  → Verify tx hash on-chain (optional sanity)    │
│          │  → Update custody_operation status              │
│          │  → Emit OFFRAMP_COMPLETED event                 │
│          │  → Notify investor                              │
└──────────┴───────────────────────────────────────────────┘
```

**Approval model:** Copper's built-in policy engine handles withdrawal approval (amount thresholds, multi-sig, whitelist-only). This is configured in Copper's dashboard. For future extensibility, Step 2 can be extended with a custom pre-approval step before the Copper submission.

### 4.4 Fund Provisioning

**Trigger:** Admin creates a new fund via API.

```
┌──────────────────────────────────────────────────────────┐
│             FUND PROVISIONING WORKFLOW                      │
├──────────┬───────────────────────────────────────────────┤
│  Step 1  │  CREATE COPPER PORTFOLIOS                      │
│          │  → Cash portfolio: "{fundName}-cash"            │
│          │  → Fund portfolio: "{fundName}-fund"            │
│          │  → Store mappings in copper_portfolios table    │
├──────────┼───────────────────────────────────────────────┤
│  Step 2  │  CREATE DEPOSIT TARGETS                        │
│          │  → Generate USDC address (Ethereum) for Cash    │
│          │  → Generate USDT address (Ethereum) for Cash    │
│          │  → Store in copper_deposit_targets table        │
├──────────┼───────────────────────────────────────────────┤
│  Step 3  │  PROVISION HARUKO                              │
│          │  → Create Haruko subfund (via @elysium/haruko)  │
│          │  → Link Copper as venue                         │
│          │  → Wait for connection                          │
├──────────┼───────────────────────────────────────────────┤
│  Step 4  │  DEPLOY ON-CHAIN                               │
│          │  → Create fund contract on blockchain           │
│          │  → Register fund in registry                    │
├──────────┼───────────────────────────────────────────────┤
│  Step 5  │  COMPLETE                                      │
│          │  → Update fund record with all IDs              │
│          │  → Emit FUND_PROVISIONED event                  │
└──────────┴───────────────────────────────────────────────┘
```

This is a **cross-system orchestration** — one workflow coordinates Copper, Haruko, and the blockchain. Step Functions makes this natural: each step calls a different service, with retries and error handling per step.

### 4.5 Wallet Registration

**Trigger:** Investor registers their wallet via API.

```
1. API receives wallet address + network from investor
2. Store in investor_wallets table (verified = false)
3. Register in Copper Address Book (for future withdrawals)
   → Store copper_address_id mapping
4. Mark verified = true
5. Emit WALLET_REGISTERED event
```

This is a simple sequential operation — no Step Function needed. A single Lambda behind the API handles it.

### 4.6 Reconciliation

**Trigger:** EventBridge Scheduler, daily at 06:00 UTC.

```
For each fund:
  1. Get Copper balances (Cash + Fund portfolios) via API
  2. Get expected balances from custody_operations table
  3. Get on-chain token balances (for cross-check)
  4. Compare:
     - Copper balance == sum of confirmed operations?
     - Copper balance == on-chain balance? (for DeFi-bridged assets)
  5. If mismatch:
     → Emit RECONCILIATION_MISMATCH event (critical alert)
     → Log discrepancy details
  6. If match:
     → Emit RECONCILIATION_PASSED event
     → Log confirmation
```

## 5. Webhook Handler (Detail)

The webhook Lambda is the entry point for all Copper-initiated events. It must be fast, secure, and reliable.

### 5.1 Flow

```
API Gateway (POST /webhooks/copper)
    │
    ▼
Lambda: copper-webhook-handler
    │
    ├── 1. Extract signature header
    ├── 2. Verify HMAC-SHA256 (reject if invalid → 401)
    ├── 3. Parse event type
    ├── 4. Check idempotency (eventId in DynamoDB)
    ├── 5. Emit EventBridge event
    │       source: "elysium.custody-ops"
    │       detailType: maps from Copper event type
    ├── 6. If pending taskToken exists for this copperOrderId:
    │       → Call sfn.sendTaskSuccess(taskToken, payload)
    └── 7. Return 200 OK
```

### 5.2 Event Type Mapping

| Copper Event | EventBridge detailType | Step Function Action |
|-------------|----------------------|---------------------|
| `deposit.confirmed` | `DEPOSIT_DETECTED` | Start deposit-processing workflow |
| `order.processing` | `ORDER_PROCESSING` | (informational, logged) |
| `order.executed` | `ORDER_EXECUTED` | (informational, logged) |
| `order.settled` | `ORDER_SETTLED` | sendTaskSuccess for waiting workflow |
| `order.failed` | `ORDER_FAILED` | sendTaskFailure for waiting workflow |
| `order.cancelled` | `ORDER_CANCELLED` | sendTaskFailure for waiting workflow |

### 5.3 Idempotency

Copper may deliver webhooks more than once. The handler uses DynamoDB (TTL = 24h) to deduplicate:

```typescript
// DynamoDB: elysium-webhook-dedup
// Key: { eventId: string }
// TTL: expiresAt (24 hours from now)

const exists = await dynamodb.getItem({ Key: { eventId } })
if (exists) {
  logger.info('Duplicate webhook, skipping', { eventId })
  return { statusCode: 200 }
}
await dynamodb.putItem({
  Item: { eventId, expiresAt: Math.floor(Date.now() / 1000) + 86400 }
})
```

### 5.4 Security

- API Gateway endpoint with no authentication (Copper can't attach tokens) — verification is via HMAC signature
- IP allowlist if Copper publishes source IPs (check with Copper during onboarding)
- WAF rate limiting on the webhook endpoint (prevent abuse)
- Webhook secret stored in Secrets Manager: `/${stage}/copper/webhook-secret`

## 7. Security

### 7.1 Secrets

All Copper credentials stored in Secrets Manager:

```
/${stage}/copper/api-key          — Copper API key
/${stage}/copper/api-secret       — HMAC signing secret (NEVER logged, NEVER in env vars)
/${stage}/copper/webhook-secret   — Webhook verification secret
```

Lambda functions access secrets at cold start via IAM policy:
```json
{
  "Effect": "Allow",
  "Action": "secretsmanager:GetSecretValue",
  "Resource": "arn:aws:secretsmanager:*:*:secret:/${stage}/copper/*"
}
```

### 7.2 Network

- Copper API calls go through the existing VPC NAT Gateway (outbound HTTPS)
- Webhook endpoint on API Gateway (public, secured by HMAC verification)
- All internal communication stays within VPC

### 7.3 Withdrawal Safety

**Layer 1 — Copper-native policies** (configured in Copper dashboard):
- Whitelist-only: withdrawals can only go to Address Book addresses
- Amount thresholds: require additional approval above configurable limits
- Time delays: optional cooldown period for new addresses

**Layer 2 — Elysium validation** (in offramp workflow Step 1):
- Check investor status (active, KYC'd, not frozen)
- Check daily/monthly withdrawal limits per investor
- Check fund-level withdrawal restrictions

**Layer 3 — Future extensibility hook:**
- The offramp workflow can add a `CUSTOM_APPROVAL` step before submitting to Copper
- This step can call any approval logic (multi-sig, compliance officer review, etc.)
- Initially this step is a pass-through; activated when custom policies are needed

### 7.4 Audit Trail

Every custody operation is recorded in three places:
1. **Step Functions execution history** — full state machine trace, 90-day retention
2. **CloudWatch structured logs** — all Lambda invocations with correlationId
3. **PostgreSQL `custody_operations` table** — queryable business record with status history

The `custody_operations` table is append-only for the application role (INSERT only, no UPDATE/DELETE grants). Status changes create new rows or use a JSONB `status_history` array.

## 10. Extensibility Points

### 10.1 Adding New Custodians

The architecture is designed so Copper-specific logic is isolated:
- `packages/copper/` — only Copper API knowledge
- Workflow steps reference `packages/copper` functions, not raw HTTP
- To add a new custodian: create `packages/{custodian}` with the same interface pattern, wire up new workflow steps

### 10.2 Adding New Asset Types

Currently USDC/USDT. To add a new asset:
1. Create deposit target in Copper for the new currency
2. Add currency to the validation whitelist in deposit/offramp workflows
3. No code changes needed — the SDK functions accept any currency string

### 10.3 Custom Approval Policies

The offramp workflow has a designated extensibility point (Section 4.3, Layer 3). To add custom approval:
1. Add a `CUSTOM_APPROVAL` step to the offramp Step Function
2. This step can call any Lambda (compliance check, multi-sig, manager approval queue)
3. Uses wait-for-callback pattern if human approval is needed

### 10.4 Multi-Network Support

Copper supports multiple blockchain networks. The schema already tracks `network` per deposit target and address. To add a new network:
1. Create deposit targets for the network in the provisioning workflow
2. The SDK already passes `network` as a parameter — no code changes

## Appendix: Copper API Error Codes

| HTTP Status | Meaning | Our Response |
|-------------|---------|-------------|
| 400 | Bad request (invalid params) | Log and fail — don't retry (code bug) |
| 401 | Auth failure (bad HMAC) | Alert immediately — credential issue |
| 403 | Forbidden (policy rejection) | Fail workflow, notify admin |
| 404 | Resource not found | Fail step, log context |
| 429 | Rate limited | Retry after `Retry-After` header |
| 500 | Copper internal error | Retry with backoff (Step Functions handles) |
| 503 | Copper maintenance | Retry with backoff, circuit breaker if persistent |
